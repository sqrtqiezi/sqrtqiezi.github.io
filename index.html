Laravel 简史

发表于 2017-01-13   |  

原文地址：[https://medium.com/vehikl-news/a-brief-history-of-laravel-5d55970885bc]

在 Vehikl，我们的绝大多数 Web 应用开发是使用 Laravel PHP 框架。我们长久以来一直是 Laravel 和开源社区的一部分，通过制作包、志愿时间和支持会议，积极地贡献。

很久以前…

直到大约4年前，PHP 社区还是一个框架相互竞争的荒原。会议中充斥着各种框架的作者，讨论他们正在做什么和为何他们的框架是最好的，吸引人们参与谁的框架是最佳解决方案的吹嘘竞赛之中。

而今天，社区变的更有凝聚力和协作性， Laravel 也在这场运动中演进。

Laravel 全景

Taylor Otwell， 来自阿肯色州的一名 .net 开发者，在使用 CodeIgniter 时萌发了创造 Laravel 的种子。他说：“我无法在没有破坏框架内部代码的情况下，添加我想要的所有功能”。他想要更简洁，更简单，更灵活。

这些心愿，加上 Taylor 的 .net 背景，促生了 Laravel 框架。

传统上，PHP 一直像编程语言的狂野西部。因为它是动态类型，在很多可能会出错的事情上没有警告。举个例子，在其他语言中，代码以这样的方式编写：如果有错误，就在错误代码执行之前捕获它。不管是尝试使用错误代码还是编译它，编译器都会阻止你。在代码中识别出问题，并要求你解决它才可以继续。

开箱即用的 PHP 没有这一切。所以理论上，你可以持续编写不良代码，而对它们是否工作却永远一无所知。当一些事情出了问题时，你必须花费很多的时间来探查问题。

Taylor 旨在通过 Laravel 来解决这些问题。他借鉴了 .net 基础设施，这些基础设施由微软建立，并且花费了数亿美元用于研究。

Taylor 寻求创建一个以简单性而闻名的框架，这种简单性表现为富有表达力的语法，清晰的结构，和非常详尽的文档。于是，Laravel 诞生了。

Laravel 基础设施

他的工作起始于一个简单的路由层，一个真正简单的控制器类型接口（model， view，controller 或简称 MVC）；v1 和 v2 分别于2011年6月和2011年9月发布，间隔仅几个月。仅过了一年之后的2012年2月， Laravel 3 发布，这时候 Laravel 的用户群和人气开始了真正的增长。

在2013年5月，Laravel 4 作为一个被完全重写的框架发布，包含一个名为 Composer 的包管理器。Composer 是一个应用级的包管理器，它将人们从相互竞争导向相互协作。

在 Composer 出现以前，没有办法通过使用两个独立包里各自的一些代码片段，来创建一个单一的解决方案。首先你需要下载这些文件，放到指定的位置，搜寻使用它们的方法，等等一堆繁琐的事情。

Composer 成为一个零界点，突然所有的框架作者开始关注彼此，开始看到协作而不是竞争的好处，并且 Laravel 开始在 PHP 世界中产生真正的影响。今天，我们正在使用的 Laravel 5.2 ，来自 Taylor 对这个框架的持续改进和创新。

牛仔的套索

随着这个单词的传播，很多西部狂野风格的 PHP 程序员开始注意到 Laravel。人们开始意识到 Laravel 非常的结构化，并且语法友好，富有表达力，组件很好的被命名。Taylor 的目标是创建一个框架，可以让最不熟练的程序员从无到有的创建应用，并使启动一个在线应用的体验让人感到愉悦。

著名 PHP 开发和 Laravel 布道者 Jeffrey Way 说，他对 Laravel 有种“直觉”。Taylor 的“天性是完全不同的，”他说：

“我有明显的印象，他有其他框架／库的作者没有的一些理解：最终，Laravel 是一个产品。这意味着代码虽然很重要，但是开发者的幸福、易用性、灵活性和品牌也同样重要… Laravel 让编码变的好玩，而无论你技能水平的高低。”

Laravel 的座右铭之一是“开发人员从下载到部署的幸福”。伴随 Laravel 到目前为止的生命周期，Taylor 和开源社区不断更新和改进框架，并且依然可以免费使用。

一个巨大并且充满激情的社区也伴随框架不断成长，Vehikl 也是这个社区充满自豪的一个部分。我们不仅使用 Laravel 为我们自己和我们的客户构建应用，我们在开源社区保持活跃，享受支持，参加和赞助会议。包括 Laravel 的官方会议， Laracon US 和 Laracon EU。Laracon US 起始于 2013年，起先只有 70 个参会者，在2015年到2016年已经增长到500人， 这反映了 Laravel 和它的社区的成长。

PHP是如何执行的 - 从源代码到渲染

发表于 2017-01-13   |  

原文地址：[https://www.sitepoint.com/how-php-executes-from-source-code-to-render/]

简介

当我们执行一段 PHP 代码时，会发生很多事情。一般来说，PHP 解释器执行代码时会经历四个阶段：

  1. 词法分析
  2. 语法分析
  3. 汇编
  4. 解释执行

第一阶段 - 词法分析

词法分析（或者说符号化）是将字符串（这里特指 PHP 源代码）转换成一系列单词符号（Token）的过程。Token 只是它所匹配的值的命名标识符。PHP 使用 re2c 从 zend_language_scanner.l 定义文件生成词法分析器。

我们可以使用 tokenizer 扩展来查看词法分析阶段的输出：

输出为：

上面输出有几个需要注意的地方：第一，并不是所有的源代码片段都会被命名为 token ，相反的，一些符号被认为是它们自己的 token （例如 =, ;, :, ?, 等）。第二，词法分析器实际上所做的不仅仅是输出一个 token 流。在大多数情况下，它还存储词法（ token 所匹配的值）和匹配 token 的行号（用于堆栈跟踪）。

第二阶段 - 语法分析

语法分析器也是用工具生成的，这次用 Bison 通过 BNF 语法文件 生成。PHP使用 LALR(1) （look ahead，left-to-right）上下文无关语法。look ahead 可以简单理解为，语法分析器可以读取 n （当前的情况下，n为1）个 token ，以解析在分析时遇到的模糊性。left-to-right 的意思是从左到右分析 token 流。

语法分析器生成之后，从词法分析器接收 token 流作为输入，并完成两项工作。首先，它尝试将 token 顺序与 BNF 语法文件中定义的任何一条语法规则进行匹配，以验证有效性。这确保了 token 流中的 token 所形成的语言构造是有效的。语法分析器的第二项工作是生成抽象语法树（AST） - 这将作为下一阶段（汇编）所使用的源代码视图树。

我们可以借由 php-ast 扩展查看由语法分析器所生成的 AST 形式。内部 AST 不会直接暴露，因为它不是很“清晰”（在一致性和一般可用性方面），因此 php-ast 扩展执行了一些转换，使其更好地被我们查看。

树节点（通常为 ast\Node 类型）具有多个属性：

  1. kind - 描述节点类型的整型；每个具有相应的常数值（例如：AST_STMT_LIST => 132, AST_ASSIGN => 517, AST_VAR => 256）
  2. flags - 指定重载行为的整型（例如：ast\AST_BINARY_OP 会有一个 flags 以区分当前发生的是哪一个二元操作）
  3. lineno - 行号，从前面的 token 信息可以看出
  4. children - 子节点，通常是进一步细分的节点（例如函数节点会有如下子节点：参数、返回值、函数体等）

该阶段输出的 AST 对于诸如Phan 之类的静态代码分析工具是极为有用的。

第三阶段 - 汇编

汇编阶段使用 AST，它通过递归遍历树发出指令码。此阶段还执行一些优化操作。包括使用字面参数（例如把 strlen(“abc”) 变成 int(3)）和折叠一些常量数字表达式（例如把 60 * 60 * 24 变成 int(86400)）来处理函数调用。

第四阶段 - 解释执行

最后一个阶段是操作码的解释。这是操作码在 Zend Engine (ZE) VM 上执行的阶段。实际上这个阶段很少被提及（至少从高级视角来说）。输出几乎任何你通过 PHP 脚本命令的输出，例如：echo，print，var_dump，等等。

因此，与其从这个阶段挖掘复杂的东西，不如关注一个有趣的事实：PHP 生成它自己的虚拟机时，需要依赖它自身。这是因为通过 PHP 脚本生成 VM，更易于编写与维护。

结尾

我们简单介绍了 PHP 解释器在执行 PHP 代码时经历的四个阶段。并涉及了使用各种扩展（包括tokenizer, php-ast, OPcache, and VLD）来操控和查看每个阶段的输出。

我希望这篇文章帮助您更好的从整体上理解 PHP 解释器，并且展现出 OPcache 扩展（它的缓存和优化能力）的重要性。

还有 monitor:delete， monitor:enable，和 monitor:disable 这些命令。第一个用于从监控列表中删除网站，其他两个命令用于启用和禁用所监控的网站：

综述

如果你打开配置文件 config/laravel-uptime-monitor.php，你会看到 Laravel Uptime Monitor有很多配置项，是高度可定制的。请注意如下的重要选项：

  • 首先，你可以自定义通知选项，为任何 Laravel Uptime Monitor 软件包可能触发的事件（它提供六种不同的事件类型）添加你的通知频道。你可以轻松的使用任何 Laravel 5.3 通知系统支持的频道，例如 Slack 或者邮件等。
  • 如果你在多个地区使用 Laravel Uptime Monitor ，你可以使用 location 来区分不同的地区，location 会和每个消息通知一起发送给你。
  • resend_uptime_check_failed_notification_every_minutes 选项用于指定你的网站关闭时，发出通知的间隔时间，默认为 60 分钟。
  • 接下来的两个选项，一个用于指定通知的邮件，另一个用于如果你使用了 Slack 通知频道时，指定的 webhook_url。
  • run_interval_in_minutes 用于指定两次运行检查的间隔时间。注意，如果运行检查失败，软件包将忽略此选项，并且开始每分钟检查该站点。
  • timeout_per_site 用于指定在检查过程中放弃之前所需等待的时间，默认为10秒。
  • fire_monitor_failed_event_after_consecutive_failures 用于指定触发 Spatie\UptimeMonitor\Events\MonitorFailed 事件之前，尝试运行检查失败的次数，默认值为 2。
  • fire_expiring_soon_event_if_certificate_expires_within_days，当 SSL 证书中剩余的天数少于你为此选项指定的天数时，将触发 SslExpiresSoon 事件。
